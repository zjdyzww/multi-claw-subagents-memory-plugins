# Memory Manager Skill v2 — 记忆管理器（协同版）

> 核心改进：适配 System 2 海绵式 / System 1 淘金式框架，保证三地一致性

---

## 核心定位

`memory-manager` 是 System 1 淘金式提炼的执行者：
- 接收 System 2 的记忆表象
- 按照提炼标准筛选有效事实点
- 更新 MEMORY.md（金字塔 L1-L4）
- 触发 Gitea 同步

**不负责**：
- System 2 海绵式吸收（那是 dual-thinking 的职责）
- 直接响应用户消息

---

## 三地一致性模型

```
System 2 表象（会话内存）
      ↑↓
MEMORY.md（本地持久，~2200字）
      ↑↓
Gitea 仓库（远程备份，全量快照）
```

### 一致性保证协议

| 层级 | 更新优先级 | 同步时机 |
|------|-----------|---------|
| System 2 表象 | 最高（最新对话） | 实时（会话内） |
| MEMORY.md | 中（有效沉淀） | 每次淘金完成后 |
| Gitea | 低（备份/版本） | 10:00/22:00 + 重大变更立即 |

### 冲突处理

```
IF System 2 表象 与 MEMORY.md 矛盾 THEN
  → System 2 表象优先
  → MEMORY.md 必须跟随更新
  → 标记本次更新为 [一致性修复]
FI

IF MEMORY.md 与 Gitea 不同步 THEN
  → 以 MEMORY.md 为准（最新有效沉淀）
  → 下次 Gitea 同步时覆盖
FI
```

## 接收与解析：System 2 表象处理

### 输入：System 2 记忆表象格式

```markdown
## System 2 Memory表象 — {时间戳}

### [待验证事实]
- {话题}: {描述} (来源: 第N轮)

### [偏好信号]
- {偏好类型}: {描述} (来源: 第N轮)

### [环境数据]
- {环境项}: {值} (来源: 第N轮)

### [决策记录]
- {决策内容} (来源: 第N轮)

### [不确定性]
- {模糊点}: {描述}
```

### 解析步骤

```
1. 遍历 [待验证事实] → 逐条判定是否有效
2. 遍历 [偏好信号]   → 全部标记为高优先级
3. 遍历 [环境数据]   → 判定是否与 MEMORY.md L3 一致
4. 遍历 [决策记录]   → 判定是否为新决策
5. 收集 [不确定性]   → 记录残差，待后续处理
```

## 提炼标准（7条）

### 有效事实点判定

| # | 类型 | 优先级 | 写入层级 |
|---|------|--------|---------|
| 1 | 用户偏好明确表达 | 🔴 最高 | L1 |
| 2 | 环境/配置变更 | 🔴 最高 | L1 或 L3 |
| 3 | 关键决策结论 | 🟠 高 | L1 |
| 4 | 新增业务/项目 | 🟠 高 | L2 |
| 5 | 人员/团队变更 | 🟡 中 | L2/L3 |
| 6 | 约束条件明确 | 🟡 中 | L1/L3 |
| 7 | 首次获得的重要参数 | 🟡 中 | L2/L3 |

### 优先级压缩规则

当多条事实点同时竞争写入同一位置时：
- 同优先级 → 按时间顺序（最新优先）
- 不同优先级 → 高优先级覆盖低优先级
- L1 冲突 → 记录旧值为历史标注，不直接删除

## 更新协议：MEMORY.md 写入

### 写入前检查

```
[ ] 有效事实点数量？
[ ] 涉及哪些层级？（L1/L2/L3/L4）
[ ] 是否有 L1 冲突需要标注历史？
[ ] 当前 MEMORY.md 总字数？
[ ] 写入后是否超出 ~2200 字上限？
```

### 写入执行

**L1 更新**（最优先，变更即写）：
```markdown
## ◆ L1 — 核心结论

1. [变更项]: [新值]   ← 覆盖更新
   ← 旧值标注: [历史记录，待归档至 VERSION_INDEX]
```

**L2 更新**（次优先）：
```markdown
### 主线A: [名称]
- 目标: [更新]        ← 局部更新
- 当前状态: [变更]
```

**L3/L4 更新**（按需）：
```markdown
## L3/L4
- [变更项]: [新值]
```

### 总量控制

| 当前字数 | 策略 |
|---------|------|
| < 2000 | 正常写入 |
| 2000-2200 | 精简 L3/L4 表述后写入 |
| > 2200 | 优先压缩 L4（外置文件索引） |
| > 3000 | 拆分到独立文件，MEMORY.md 只保留 L1-L2 |

## 同步协议：Gitea 仓库

### 同步触发条件

| 条件 | 触发时机 |
|------|---------|
| 定时 | 每天 10:00 和 22:00 |
| 重大变更 | 战略方向/核心配置变更，立即同步 |
| 一致性修复 | System 2 与 MEMORY.md 冲突后立即同步 |
| System 2 启动 | 拉取 Gitea 最新到本地 |

### 同步操作序列

```bash
# 1. 读取当前 MEMORY.md
cat ~/.hermes/memories/MEMORY.md

# 2. 评估是否需要写入
# （如果 System 2 有新事实点，则写入）

# 3. Git 操作
cd ~/.hermes/memory-agent  # Gitea 仓库本地克隆
git add -A
git commit -m "Memory sync $(date '+%Y-%m-%d %H:%M')
- 新增: [L1变更点]
- 更新: [L2变更点]
- 一致性修复: [如有]
- 残差不确定性: [数量]"
git push origin main

# 4. 验证推送
git log --oneline -1
```

### 版本索引更新

每次 Gitea 同步后，更新 `VERSION_INDEX.md`：

```markdown
## v{N+1} — {日期} {时间}

### L2 变更
- [变更内容]

### AI 标签
- [战略] / [配置] / [偏好] / [项目] / [人员]

### 残差不确定性
- [数量] 条，待后续处理
```

## 残差不确定性管理

```markdown
## 残差记录 — {时间戳}

| # | 不确定项 | 描述 | 状态 |
|---|---------|------|------|
| 1 | ... | ... | 待验证 |

```

下次对话优先处理残差，优先于新的 [不确定性]。
