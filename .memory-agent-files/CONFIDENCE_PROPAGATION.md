# 置信度传播机制

> 版本: v1 | 2026-05-07 | P1优化
> 目标: 存储时固化置信度元数据，可追溯每个事实的来源和验证时间

---

## 置信度等级

| 等级 | 标识 | 定义 | 写入位置 |
|------|------|------|---------|
| 高 | 🟢 | 用户明确表达 / 多次验证一致 | L1/L2 |
| 中 | 🟡 | 单次获取 / 推断得出 / 需验证 | L2/L3 |
| 低 | 🔴 | 模糊表达 / 单次信号 / 可能有误 | L3 |

---

## 置信度元数据结构

```markdown
## [事实内容]
- **来源**: {来源描述}
- **置信度**: {🟢🟡🔴}
- **验证时间**: {YYYY-MM-DD}
- **替代记录**: [{旧值}] → [{时间}]  # 可选，已被覆盖的旧值
- **备注**: {补充说明}  # 可选
```

---

## 置信度判定规则

### 🟢 高置信度 (直接写入 L1)

满足任一:
1. 用户明确表达的事实（"我的微信是XXX"）
2. 环境/配置变更（用户确认）
3. 关键决策结论（用户明确同意）
4. 多轮验证一致的事实

### 🟡 中置信度 (写入 L2/L3)

满足任一:
1. 单次获取的信息（首次对话）
2. 基于上下文的合理推断
3. 用户态度模糊（"可能吧"）
4. 需要后续验证的假设

### 🔴 低置信度 (不写入或写入 L3 末尾)

满足任一:
1. 模糊表达（"大概", "可能"）
2. 单一信号来源
3. 与已知事实有矛盾
4. 用户自己也未确认

---

## 置信度更新协议

```
收到新信息时:

IF 新信息 vs 已有记忆:
  
  CASE 1: 新信息置信度 > 已有置信度
    → 替换已有记忆
    → 记录替代记录: [旧值] → [新值 @ 时间]
    → 更新置信度为新值
  
  CASE 2: 新信息置信度 == 已有置信度
    → 冲突标记
    → 记录矛盾: 需要用户确认
    → 保持两者（带冲突标记）
  
  CASE 3: 新信息置信度 < 已有置信度
    → 忽略新信息
    → 记录: [忽略原因] @ 时间
```

